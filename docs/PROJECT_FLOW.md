# FlowTrace 项目流程

本文档用于固化 FlowTrace 的产品主流程，防止后续开发在局部 bug、临时 UI 或单个示例项目中跑偏。

## 一句话目标

FlowTrace 的目标不是单纯做日志 viewer，也不是让 LLM 自动猜项目逻辑，而是建立一个本地参数流观测闭环：

```text
读取用户项目 -> 建立项目结构候选图 -> 用户运行真实/模拟流程 -> 采集参数传递事实 -> 对齐静态结构与运行事实 -> 可视化定位问题
```

## 总流程

### 1. 初始化项目

输入：

```text
用户项目路径
可选配置文件
```

输出：

```text
FlowTrace 项目上下文
默认配置
本地 .flowtrace 目录
```

本阶段必须回答：

```text
当前分析的是哪个项目？
项目根目录在哪里？
运行记录写到哪里？
哪些目录应该扫描？
哪些目录必须排除？
```

### 2. 读取用户项目

输入：

```text
项目根目录
扫描配置
```

输出：

```text
project model
```

第一版 project model 至少包含：

```text
modules
classes
functions
entrypoint_candidates
imports
declared_methods
```

这一阶段只建立候选结构，不宣称已经理解真实 workflow。

### 3. 识别候选 workflow

输入：

```text
project model
```

输出：

```text
候选入口
候选调用边
候选层级
未覆盖方法目录
```

规则：

```text
静态扫描只产生候选。
运行事实才能验证候选。
LLM 只能辅助解释候选。
```

### 4. 运行真实或模拟流程

输入：

```text
用户操作
CLI 命令
测试命令
HTTP 请求
示例模拟输入
```

输出：

```text
run
trace
span
event
```

本阶段重点不是模拟一切，而是允许用户用最接近真实业务的方式触发流程。

### 5. 采集参数传递事实

输入：

```text
被 trace_node / adapter 包裹的方法调用
```

输出：

```text
输入快照
输出快照
异常
耗时
契约校验
字段 diff
调用上下文
```

采集层必须保持稳定、可预测、低侵入。

### 6. 对齐静态结构与运行事实

输入：

```text
project model
runtime events
```

输出：

```text
已覆盖方法
未覆盖方法
已验证调用边
仅静态候选边
仅运行时发现边
问题列表
```

这一步是 FlowTrace 区别于普通日志工具的关键。

### 7. 可视化定位问题

输入：

```text
分析结构
数据流
层级流
问题列表
运行对比
监视项
```

输出：

```text
运行视图
项目结构视图
层级流转视图
方法数据流视图
全量细节视图
问题列表
监视窗口
事件详情
```

前端只负责表达和交互，不负责重新推断业务事实。

### 8. 导出与复现

输入：

```text
run_id
project model
runtime events
analysis output
```

输出：

```text
可提交给 Codex 阅读的记录包
```

导出必须做到：

```text
结构完整
字段脱敏
版本明确
可离线阅读
不需要猜测上下文
```

## 数据优先级

当不同来源结论冲突时，优先级固定为：

```text
运行事实 > 确定性静态扫描 > 用户配置 > LLM 辅助解释
```

LLM 不得作为最终事实来源。
