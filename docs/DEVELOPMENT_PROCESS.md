# FlowTrace 开发流程

本文档用于固化 FlowTrace 后续开发和修 bug 的流程，避免通过不断新增层级来掩盖边界不清的问题。

## 开发原则

### 1. 先归因，再修改

每个问题必须先归类：

```text
接入问题
上下文问题
采集问题
契约问题
存储问题
扫描问题
分析问题
API 问题
UI 问题
诊断问题
导出问题
```

归类后，只在对应层级内修复。

### 2. 默认不新增层级

修 bug 时禁止第一反应是：

```text
再加一个管理层
再加一个中间层
再加一个协调器
再加一个兜底模块
```

这会让项目逐渐变成不可理解的补丁集合。

### 3. 新功能先找边界

任何新功能必须先写清楚：

```text
输入
输出
归属层级
影响哪些 API
影响哪些 UI
是否影响事件 schema
是否影响旧运行记录
是否需要测试夹具
```

写不清楚则先不实现。

### 4. 运行事实优先

当 UI、扫描结果、LLM 解释和运行记录冲突时：

```text
运行记录优先。
```

FlowTrace 的核心不是猜测，而是帮助用户看见真实参数传递。

## 任务进入流程

每个任务进入开发前，先执行以下判断：

```text
1. 这是 bug、功能、重构、文档还是验证？
2. 属于哪个层级？
3. 是否改变数据 schema？
4. 是否影响旧运行记录？
5. 是否需要前端状态迁移？
6. 如何验证？
```

## Bug 修复流程

### 第一步：复现

必须尽量拿到：

```text
用户操作
运行记录
报错信息
截图
相关 API 输出
相关前端状态
```

如果无法复现，也要记录当前证据。

### 第二步：归属

按 `ARCHITECTURE_LAYERS.md` 的 bug 归属表定位层级。

示例：

```text
边线显示错误 -> Analysis 或 UI Graph
方法没有出现在图中 -> Scanner 或 Analysis
监视窗口滚动丢失 -> UI
运行记录缺字段 -> Instrumentation 或 Storage
```

### 第三步：层内修复

只改归属层和必要调用边界。

避免：

```text
为了修 UI bug 修改存储格式。
为了修扫描 bug 改前端特殊判断。
为了修分析 bug 增加一层临时适配器。
```

### 第四步：补验证

按问题类型选择：

```text
JS 语法检查
Python compileall
API 请求检查
静态文本检查
路由样例检查
JSONL 夹具快照
浏览器截图检查
```

### 第五步：写记录

需要写入：

```text
docs/OPTIMIZATION_LOG.md
D:\DEBUG\PROJECT_DEVELOPMENT.md
```

记录应包含：

```text
问题
根因
改动
验证
剩余风险
```

## 新功能开发流程

### 第一步：放入流程位置

每个功能必须属于主流程之一：

```text
初始化项目
读取项目
识别候选 workflow
运行流程
采集参数
分析结构
前端定位
导出复现
诊断恢复
```

不属于这些流程的功能，默认推迟。

### 第二步：确定归属层

示例：

```text
项目结构扫描 -> Project Scanner
覆盖率对齐 -> Analysis
新增接口 -> Service API
新增视图 -> UI
启动检查 -> Diagnostics
```

### 第三步：定义结构

需要写清楚标准输入/输出。

例如 project scanner 的输出不能直接服务某个 UI，而应是标准 project model。

### 第四步：实现最小闭环

先实现端到端最小闭环，再扩展细节。

例如项目读取层 MVP：

```text
扫描 Python 文件
输出模块/类/函数/入口候选
API 返回 project model
前端显示项目结构
```

不要第一版就做全自动 workflow 理解。

## 重构流程

允许重构的情况：

```text
职责明显混杂
同一逻辑重复三次以上
已有测试或夹具能保护行为
用户目标需要更稳定边界
```

不允许重构的情况：

```text
只是代码看起来不舒服
只是为了套某个架构模式
只是为了修一个局部 bug
没有验证手段
```

## 新增层级流程

新增层级必须先完成文档变更：

```text
1. 在 ARCHITECTURE_LAYERS.md 写明新层级职责。
2. 写明它的输入、输出、不负责事项。
3. 写明依赖方向。
4. 写明至少两个调用方或复用场景。
5. 写明测试方式。
```

没有完成以上内容，不允许新增层级。

## 验收门槛

一个阶段完成必须满足：

```text
功能闭环存在。
数据结构稳定。
文档记录完整。
至少有命令级验证。
没有明显跨层职责污染。
```

前端功能额外要求：

```text
不破坏现有主流程。
不让 UI 承担业务分析。
不通过硬编码掩盖数据问题。
```

项目读取层额外要求：

```text
扫描结果可复现。
扫描结果与运行事实分离。
候选关系不得伪装成已验证关系。
```

```text
项目流程梳理与项目读取层 MVP
```

执行顺序：

```text
1. 固化项目流程文档。
2. 固化架构层级边界。
3. 固化开发流程。
4. 设计 project model。
5. 扩展 scanner.py。
6. 增加 project API。
7. 增加项目结构/覆盖视图。
8. 再进入完整工程化部署。
```
