# FlowTrace MVP 项目说明书

本文说明 FlowTrace MVP 前端的启动方式、真实项目接入流程、各模块用途，以及常见问题处理。

## 1. 启动与访问

推荐使用仓库根目录的启动脚本：

```bat
cd /d D:\pyProject\flowtrace-mvp
start_flowtrace.bat
```

也可以手动启动：

```bat
cd /d D:\pyProject\flowtrace-mvp
set PYTHONPATH=D:\pyProject\flowtrace-mvp\src
.\.venv\Scripts\python.exe -m flowtrace.server
```

服务启动后，在浏览器访问：

```text
http://127.0.0.1:8765/
```

## 2. 前端整体用途

FlowTrace MVP 是一个面向项目运行链路观察、只读结构扫描、运行记录查看、问题定位和受控 Runner 执行的治理控制台。

前端不会直接修改目标项目代码。它主要读取：

- 目标项目目录；
- FlowTrace 运行记录目录，默认是目标项目下的 `.flowtrace`；
- 可选配置文件；
- Runner 工作台中明确确认后的低风险样例执行结果。

真实执行能力目前仍受治理限制，默认只按低风险样例项目推进，不建议直接对生产项目做真实执行。

## 3. 接入真实项目

进入 `项目结构` 页签，在顶部的 `项目接入` 区域完成接入。

操作步骤：

1. 点击 `项目文件夹` 后面的 `选择文件夹`。
2. 在系统文件夹选择器中选择完整项目根目录。
3. `运行记录目录` 会默认同步为 `<项目根目录>\.flowtrace`。
4. 如需读取其他运行记录目录，点击 `运行记录目录` 后面的 `选择文件夹` 手动选择。
5. 如需指定配置文件，点击 `配置文件` 后面的 `选择文件`。
6. 点击 `应用项目`。

接入后，FlowTrace 会切换当前读取上下文，并刷新项目结构、运行记录、问题列表等视图。

如果系统文件夹选择器没有弹出，请检查它是否被其他窗口遮挡。也可以直接把完整路径粘贴到输入框。

## 4. 页签模块说明

### 项目结构

用于查看当前目标项目的静态结构与接入状态。

主要内容：

- `项目接入`：选择真实项目文件夹、运行记录目录和可选配置文件。
- `项目概览`：显示当前项目根目录、运行记录来源、扫描状态。
- `入口与模块`：展示识别到的入口文件、模块和项目结构。
- `扫描问题`：展示静态扫描阶段发现的问题，例如不可读路径或解析失败文件。

适合用于第一步确认：FlowTrace 是否已经读取到正确项目。

### 接入向导

用于把目标项目整理成可治理、可观测的运行上下文。

主要内容：

- 检查项目路径、运行记录路径和配置路径是否合理。
- 保存或更新运行画像。
- 给后续 Runner 工作台提供受控执行所需的上下文。

适合用于从“只读观察”过渡到“受控执行准备”。

### 层级流转

用于查看一次运行中的层级调用关系。

它帮助判断请求或任务在入口层、服务层、工具层、外部依赖层之间如何流转。

适合用于回答：

- 调用从哪里进入；
- 中间经过哪些层；
- 是否出现跨层跳转或异常路径。

### 方法数据流

用于查看方法级别的数据流转。

它关注函数、方法、参数、返回值和关键数据对象之间的关系。

适合用于回答：

- 某个数据从哪里来；
- 经过哪些方法处理；
- 最终流向哪里；
- 是否存在明显的数据断点。

### 全量细节

用于查看当前运行记录的完整结构化细节。

该页签更接近底层记录，信息密度较高，适合排查其他视图中无法解释的细节问题。

### 问题列表

用于集中查看 FlowTrace 在扫描、运行记录分析或治理检查中发现的问题。

常见问题包括：

- 缺少运行记录；
- 项目结构无法完整解析；
- Runner 前置条件不足；
- 某些事件缺少必要字段。

适合用于收敛下一步修复方向。

### 运行对比

用于比较两次运行记录之间的差异。

适合用于回答：

- 本次运行比上次多了什么事件；
- 调用路径是否发生变化；
- 问题是否复现；
- 修复后链路是否变短或变稳定。

### 事件流程

用于按时间顺序查看事件流。

它强调事件发生顺序、状态变化和关键节点，适合排查异步流程、长链路流程和运行中断点。

### Runner 工作台

用于受控启动低风险样例项目，并把启动、预检、确认、快照、执行和停止统一纳入治理流程。

Runner 工作台包含：

- 运行画像：目标项目、启动命令、端口、健康检查等信息。
- 预检：确认路径、命令、端口、依赖和风险边界。
- Gate：确认是否满足真实执行前置条件。
- 会话：显示当前 Runner 会话状态。
- 快照：保存执行前状态，便于审计。
- Dry-run：先模拟执行，不启动真实目标。
- 真实执行：仅在满足治理条件和显式确认后启动。
- 停止控制：对运行中的目标进行取消或超时处理。

当前真实执行仍建议限定在 `examples\realistic_projects` 下的低风险样例项目。

## 5. Runner 确认口令

Runner 工作台中部分操作需要显式确认，以避免误触发真实执行。

常见确认文本：

```text
RUN TARGET PROJECT
```

用于确认启动目标项目。

```text
CONTROL TARGET PROJECT
```

用于确认停止、取消或超时控制目标项目。

如果确认文本不匹配，前端不会继续执行对应动作。

## 6. 推荐使用流程

### 只读接入与观察

1. 打开 `项目结构`。
2. 选择真实项目文件夹。
3. 保持运行记录目录为默认 `.flowtrace`，或选择已有记录目录。
4. 点击 `应用项目`。
5. 查看 `项目结构`、`问题列表`、`事件流程` 和 `全量细节`。

这是接入真实项目的最低风险方式。

### 分析一次已有运行

1. 确认运行记录目录中已有 FlowTrace 记录。
2. 在左侧或运行记录选择区域刷新运行记录。
3. 打开 `层级流转`、`方法数据流`、`事件流程`。
4. 用 `问题列表` 收敛异常点。
5. 用 `运行对比` 比较修复前后的两次记录。

### 准备低风险真实执行

1. 先在 `项目结构` 接入项目。
2. 到 `接入向导` 保存运行画像。
3. 到 `Runner 工作台` 执行预检。
4. 通过 Gate 检查。
5. 创建执行前快照。
6. 先执行 Dry-run。
7. 仅在确认是低风险样例项目时，再输入确认文本执行真实启动。

## 7. 当前边界

当前版本支持真实项目的前端接入和只读扫描，但真实执行仍有边界：

- 不会自动修改目标项目源码。
- 不建议直接对生产项目真实执行。
- Runner 真实执行优先服务低风险样例项目。
- 如果要接入真实业务项目进行 Runner 测试，建议先复制到隔离目录或样例沙箱。
- 对外部服务、数据库、队列、云资源有写入风险的项目，应先补充更严格的 allowlist、环境隔离和回滚方案。

## 8. 常见问题

### 页面打开后没有内容

先确认服务是否启动：

```bat
cd /d D:\pyProject\flowtrace-mvp
start_flowtrace.bat
```

然后打开：

```text
http://127.0.0.1:8765/
```

如果仍然空白，刷新页面或重启服务。

### 无法选择完整项目

请使用 `项目结构` 页签中 `项目文件夹` 后面的 `选择文件夹`。该入口用于选择目录，不是选择单个文件。

如果系统选择器被遮挡，可以直接粘贴项目根目录路径。

### 没有运行记录

检查 `运行记录目录` 是否指向正确目录。默认目录是：

```text
<项目根目录>\.flowtrace
```

如果目标项目之前没有生成过 FlowTrace 记录，运行记录视图会为空。

如果目标项目使用自己的 `runtime` 目录保存运行审计、报告或日志，而不是 FlowTrace 的 `.flowtrace` 格式，可以把 `运行记录目录` 指向该 `runtime` 目录。FlowTrace 会在没有原生 `runs\<run_id>\events.jsonl` 时，只读扫描前三层内的 `.json`、`.jsonl`、`.log`、`.txt` 文件，并把它们投影成外部运行记录。

外部 runtime 投影的边界：

- 可以让运行列表、事件详情和问题排查入口不再空白。
- 每个外部文件会被显示为一条文件级运行记录。
- 这是只读兼容视图，不会修改真实项目。
- 它不等同于完整方法级链路追踪；`层级流转` 和 `方法数据流` 需要目标项目写出 FlowTrace 事件，或后续增加针对该 runtime 格式的专用转换器。

### Runner 无法真实执行

常见原因：

- 未保存运行画像；
- 预检未通过；
- Gate 条件不足；
- 未创建快照；
- 未执行 Dry-run；
- 确认文本不匹配；
- 目标不是当前允许的低风险样例项目。

优先在低风险样例项目中完成闭环，再扩展到真实业务项目。

## 9. 安全建议

接入真实项目时，建议按以下顺序推进：

1. 只读接入项目目录。
2. 读取已有运行记录。
3. 分析问题列表和事件流程。
4. 在隔离副本中验证 Runner 流程。
5. 明确 allowlist、环境变量、端口和外部依赖后，再考虑更广泛的真实执行测试。

这样可以把 FlowTrace 的价值先用于观察和治理，再逐步扩大执行范围。
