
# FlowTrace

> 一个关于运行时可观测性（Runtime Observability）的探索项目。

> **项目状态：Archived（停止维护）**

---
# 一个完美的IDEA

这个项目的想法出现的十分突然：

某天，我正用着ChatGPT提供的prompt软鞭狠狠地抽打Codex的屁股，驱使着它努力干活，忽略了对代码的review，只是让Codex沿着我和ChatGPT固定的路线继续开发，后来真实输入验证的时候，Bug 开始成片地冒出来。

我挥舞着prompt让Codex自行审计，最终定位问题，发现Codex在干活的时候偷懒了！它在数据结构中添加新的参数时并没有对之前的参数进行及时更新！

为了防止以后再次出现这种低级错误，我开始设想一个自动可视化工具，也就是这个项目。

---
# 最初的设想

这个工具，它应当能够监听一个项目的全部流程，获悉用户的所有操作，无论是用户项目的主流程，还是用户切换页面的副逻辑，它都应当获取到方法之间的数据传递信息与内容。

它应当借助LLM或者用户自己使用项目提供的方法、层级节点构建项目流程，然后通过类似Unity、Unreal Engine这类工程软件的动画状态机形式对用户暴露获取到的项目各个节点之间的数据传递细节，方便用户定位问题，修复问题。

它应当帮助用户理解项目运行过程、分析数据流向、定位问题，用户应当能够像查看调用链一样查看整个业务执行流程，而不是面对大量日志和零散的运行结果。

它不对用户项目造成影响，也不应对用户项目造成影响。

# 开始实现

看着Codex上充足的plus额度与openai施舍的两次reset（甚至有两个plus账号），我信心满满，编程界的新星正在冉冉升起，天才程序员将要开始新的旅途！ 

之前Vibe Coding的经验告诉我，不能盲目相信AI，所以我一开始并没有让 Codex 按照我仅有的一个想法开始工作。

我让 Codex 先验证项目可行性，然后开始讨论技术选型，写死项目定位与目的，固定开发路径。在 Codex 输出的一份份文档报告中，自认为万无一失的我最终输入了“开始实现”的指令。

彼时我坚信这是一个能够作为我简历主打的项目，我将凭借这个项目拿到心仪的offer，接触到最新的技术。所以我在项目模拟输入 Demo 跑通之后，开始纠缠于前端的UI交互、用户体验，希望能够给使用者完美的交互体验。

类似浏览器的标签页拖拽、工程软件的工作区分布、动画状态机的流程节点连接一一实现，我的手感火热，为了前端展示绞尽脑汁，提示词也开始出现侧重。

---
# Vibe Coding —— 氛围编程

大量前端UI相关的prompt误导了 Codex ，上下文被前端内容挤满，Codex 不可避免地产生了幻觉，最终人工智能和能工智人双双陷入了幻觉，我在 Codex 营造的火热氛围中迷失，在前后端无关紧要的地方烧掉了超过3亿 token 与难以计算的时间成本，而 Codex 则在幻觉的误导下与我一起将项目推下了深渊。

直到真实输入时我才惊醒，发现项目实际上已经走到了失败的边缘。原本定位为运行时辅助工具的 FlowTrace，在 Codex 营造的氛围里逐渐演变成了一个功能复杂的工作台。前端不断塞入不必要的功能，越来越臃肿，最后光渲染就要十秒！后端则承担了越来越多本不属于它的职责，而最初希望解决的“运行时观测”问题却始终没有得到真正的解决。

FlowTrace 并没有真正观察项目。它只能根据已有文件与日志进行推断，所以我开始弥补缺失的运行信息，Projection、 Synthetic Event、 Runtime Grouping，但它们终究只是推断，不是观察。而推断，并不可靠。

接近三天，我没有让项目前进一步，只是不断让它变得更复杂。

---
# 失败的项目

发现问题之后，我尝试挽救，希望项目能够回到当初的样子，于是我开始跟 ChatGPT 反复沟通，探讨纠偏技术路线，分析如何亡羊补牢。

结论并不喜人：在不对被监控项目代码进行微调之前，很难实现用户操作级别的监控，即使结合文件监控、日志分析、代理/网关以及浏览器 DevTools 等方式，也只能重建部分运行过程，而无法获得 100% 语义级的执行链路。想要达到最初设想的效果，最终仍然需要被监控项目进行 SDK 或 Probe 接入。

这与我最初设想的“零侵入完成完整观测”的目标存在本质差异。

因此，我最终决定停止继续开发 FlowTrace，并将这个仓库保留下来，作为一次关于运行时可观测性系统设计的完整探索记录。

后来回头看，这些问题其实很早就出现了。

在我我沉迷于 UI时。

在我不断修改数据模型时。

在我没有先解决 Observation时。

在我甚至开始忘记最初想解决的问题时。

失败早已注定。

# 福报
总有人说：吃亏是福。

FlowTrace 最终没能成为我最初设想的产品。

Bug 很多。

代码债不少。

需求也没有真正解决。

但它并不是一次毫无意义的失败。

它完整验证了一套运行时可视化工具的设计思路，也让我认识到可观测性系统真正的重点不是界面，而是观测数据。

这次项目最大的价值，不是留下了一套代码，而是留下了一次完整的架构探索，以及关于可观测性设计最重要的一条经验：

> **Observation First（观测优先）永远比 Visualization First（展示优先）更重要。**
